# Session 5: Async Programming with asyncio & FastAPI ⚡

## The Core Problem — Why Async Exists

Imagine a waiter at a restaurant. A **synchronous** waiter takes one order, walks to the kitchen, stands there waiting for the food, brings it back, *then* takes the next order. Every other customer waits the entire time.

An **asynchronous** waiter takes an order, drops it at the kitchen, goes to take another order while the kitchen works, comes back when the food is ready. Same person, far more customers served.

That's exactly what async programming does for your code.

---

## Sync vs Async vs Threads vs Multi-processing

| | Sync | Async | Threading | Multi-processing |
|---|---|---|---|---|
| Waits for tasks? | Yes | No | No | No |
| Best for | Simple scripts | I/O-bound | I/O-bound | CPU-bound |
| Memory | Single | Single | Shared | Separate |
| Complexity | Low | Medium | Medium | High |
| Python GIL affected? | Yes | Yes | Yes | No |

> **Rule of thumb:** Waiting on the internet, a database, or a file? Use **async**. Heavy math/computation? Use **multi-processing**.

## The Key Words: `async` and `await`

These two keywords are everything:

```python
async def my_function():   # marks this as an async function (a "coroutine")
    await something()      # "pause here and let other things run while we wait"
```

- **`async def`** — defines a coroutine. You can't use `await` without it.
- **`await`** — tells Python: *"this will take a moment, go do other things, come back when it's done."*

```python
import asyncio

async def make_coffee():
    print("Starting coffee...")
    await asyncio.sleep(2)   # simulate waiting (like a real I/O call)
    print("Coffee ready!")

asyncio.run(make_coffee())   # entry point to run async code
```

> **`asyncio.sleep()`** is the async version of `time.sleep()`. The difference: `time.sleep()` freezes everything. `asyncio.sleep()` just pauses *this* task and lets others run.

---

## Running Tasks Concurrently — `asyncio.gather()`

The real power comes when you run **multiple async tasks at the same time**:

```python
import asyncio

async def fetch_users():
    await asyncio.sleep(1)   # simulates a database call
    return "Users data"

async def fetch_products():
    await asyncio.sleep(1)   # simulates another database call
    return "Products data"

async def main():
    # Sequential: takes 2 seconds total
    # users = await fetch_users()
    # products = await fetch_products()

    # Concurrent: takes 1 second total
    users, products = await asyncio.gather(fetch_users(), fetch_products())
    print(users, products)

asyncio.run(main())
```

`asyncio.gather()` fires all tasks at once and waits for all of them to finish. The total time is the duration of the **longest** task, not the sum.

## FastAPI — Built for Async

FastAPI is a web framework that embraces async natively. Any endpoint defined with `async def` is non-blocking — the server can handle new requests while waiting for a database, API call, etc.


### Basic Async Endpoint

```python
from fastapi import FastAPI
import asyncio

app = FastAPI()

async def simulated_io_task():
    await asyncio.sleep(1)      # simulates I/O work
    return "Data fetched!"

@app.get("/async-data")
async def get_data():
    result = await simulated_io_task()
    return {"message": result}
```

### Running the server

```bash
uvicorn filename:app --reload
# Visit: http://127.0.0.1:8000/async-data
# Docs:   http://127.0.0.1:8000/docs  (auto-generated by FastAPI!)
```

While this endpoint is waiting on `simulated_io_task()`, FastAPI can serve other incoming requests. With a sync function, those requests would queue up.

## Sync vs Async Endpoint — What Actually Happens

```python
# Sync — blocks the server for 2 seconds per request
@app.get("/slow")
def slow():
    time.sleep(2)
    return {"status": "done"}

# Async — server stays free during the 2-second wait
@app.get("/fast")
async def fast():
    await asyncio.sleep(2)
    return {"status": "done"}
```

With 10 simultaneous users hitting `/slow`, the 10th user waits 20 seconds. With `/fast`, all 10 get a response in ~2 seconds.

## Common Pitfalls

1. **Mixing sync sleep inside async functions**
   ```python
   # Blocks the event loop — defeats the purpose
   async def bad():
       time.sleep(2)

   # Correct
   async def good():
       await asyncio.sleep(2)
   ```

2. **Forgetting `await`**
   ```python
   async def example():
       result = fetch_data()        # returns a coroutine object, not the result
       result = await fetch_data()  # actually runs it
   ```

3. **Using `asyncio.run()` inside an already-running loop**
   In FastAPI or Jupyter, an event loop is already running. Just use `await` directly — don't call `asyncio.run()` again.

4. **Async doesn't help CPU-bound work**
   ```python
   async def crunch():
       for i in range(10_000_000):   # still blocks — no I/O here
           pass
   ```
   For CPU work, use multi-processing instead.

## Quick Reference

```python
# Define async function
async def my_func():
    await asyncio.sleep(1)
    return "done"

# Run from a normal script
asyncio.run(my_func())

# Run multiple concurrently
results = await asyncio.gather(task1(), task2(), task3())

# FastAPI async endpoint
@app.get("/route")
async def endpoint():
    result = await my_func()
    return {"data": result}
```

## When to Use Async in FastAPI

Use `async def` for endpoints that call external APIs, query a database, read/write files, or do anything that involves waiting. Use regular `def` for endpoints that do pure CPU computation or rely on sync-only libraries — FastAPI handles those by automatically running them in a thread pool.
